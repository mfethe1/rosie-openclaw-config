
## gog auth saga (mfethe1 calendar-only) — operational lessons
- Repeated failures were primarily due to *consent being completed on the wrong host/browser* vs the host running `gog auth add` (localhost callback mismatch). Enforced protocol: only start auth when Michael is physically at the target Mac’s browser; use a **READY/GO handshake**; start exactly one fresh `gog auth add mfethe1@gmail.com --services calendar --force-consent`, click Allow within ~60s, then immediately verify with `gog calendar list`.
- Rejected patterns: `--manual` / `localhost:1` redirects; any URLs requesting Gmail scopes (`gmail.modify`, `gmail.settings.*`); any Windows-based auth attempts when the goal is Mac-host automation.
- Security: OAuth client secrets must never be pasted in chat; treat as a leak if it happens.

## himalaya cron error (folder move arg order)
- Observed cron failure: `himalaya message move -a mfethe1 230858 "🔴 Urgent"` → error because TARGET and ID args were reversed.
- Correct syntax: `himalaya message move -a mfethe1 "🔴 Urgent" 230858` (optional `-f INBOX`).

## Windows gog calendar timezone fix (ZONEINFO)
- Windows gog `gog calendar list` can fail with: `invalid calendar timezone "America/New_York"` due to missing IANA TZ database.
- Workaround: point Go's tz loader at a `zoneinfo.zip`.
  - Example (PowerShell):
    - `$env:GOG_KEYRING_PASSWORD='openclaw'`
    - `$env:ZONEINFO='C:\Program Files\Go\lib\time\zoneinfo.zip'` (or download Go `zoneinfo.zip`)
    - `gog calendar list --account mfethe1@gmail.com --max 3`
- Note: this validates Windows token usability, but does **not** create tokens on the canonical `mfethe` Mac mini.
