## 1. Competitor Assessment: Ralph-loop vs Antfarm vs Oh-My-OpenCode

### 1.1 Ralph-loop

**Source:** https://github.com/snarktank/ralph + frankbria/ralph-claude-code variant

**What it is:**  
A shell-script autonomous loop that runs AI coding tools (Amp CLI or Claude Code) repeatedly until all PRD items are complete. Based on Geoffrey Huntley's "Ralph pattern." Each iteration spawns a fresh instance with clean context. Memory persists via `git history`, `progress.txt`, and `prd.json`.

**Architecture:**
```
PRD.md → prd.json → ralph.sh loop → [fresh AI instance per iteration] → progress.txt check → repeat until done
```

**Key features:**
| Feature | Status |
|---|---|
| Deterministic guarantees | Partial — same pattern each iteration, but LLM output is non-deterministic |
| Failure handling | Max-iterations safety limit; completion indicator detection; no automatic retry |
| Context management | **Excellent** — each iteration is a clean instance; no context bloat |
| Install/operational burden | **Minimal** — shell script + jq; no Node, no DB, no queues |
| Memory model | Git history + flat files (progress.txt, prd.json) |
| Multi-agent support | ❌ Single-agent only |
| Target platform | Claude Code, Amp CLI |
| OpenClaw integration | Partial (Claude Code variant exists; no native OpenClaw primitives) |

**Pros:**
- Dead simple: one shell script, zero dependencies
- Context window management is automatic (fresh instance each loop)
- PRD-driven: clear artifact-based continuity
- Battle-tested pattern; used in production by many teams

**Cons:**
- Single agent only — no parallel lanes, no cross-agent verification
- Memory model is very lightweight (flat files); poor for semantic retrieval
- Failure handling is limited: max_iterations stops the loop, no escalation path
- Not OpenClaw-native; requires Claude Code/Amp as underlying runner
- Not directly usable for our cron-based multi-agent team architecture

**Risk:** LOW (just a shell script)  
**Decision:** **SKIP adopting directly.** We already implement the spirit of the Ralph loop via our 3-hour cron cycles (each cycle = fresh context). Understand it as a pattern; don't add operational overhead.

### 1.2 Antfarm

**Source:** https://github.com/snarktank/antfarm

**What it is:**  
A TypeScript CLI that provisions and orchestrates teams of specialized AI agents within OpenClaw. Built ON TOP of the Ralph loop pattern, scaled to multi-agent workflows. YAML-defined deterministic pipelines with SQLite state tracking and cron coordination.

**Architecture:**
```
YAML workflow definition → antfarm provision → SQLite state DB + cron pollers → 
  [agent claims step → fresh OpenClaw session → does work → passes context → next agent] → repeat
```

**Key features:**
| Feature | Status |
|---|---|
| Deterministic guarantees | **STRONG** — same workflow = same steps = same order; not LLM-improvised |
| Failure handling | **Excellent** — automatic retry per step; exhausted retries → human escalation |
| Context management | **Excellent** — each agent gets fresh session; no 50-message-ago hallucinations |
| Install/operational burden | Medium — Node.js ≥22 required; CLI install; no Docker/Redis/Kafka |
| Memory model | SQLite for workflow state; git history for artifact continuity |
| Multi-agent support | ✅ Full — planner, developer, verifier, tester, reviewer as distinct agents |
| Target platform | **OpenClaw native** |
| Agent verification | Cross-agent: developer ≠ their own reviewer |

**Built-in workflows:**  
- `feature-dev` (7 agents): plan → setup → implement → verify → test → PR → review  
- `security-audit` (7 agents): scan → prioritize → setup → fix → verify → test → PR  
- `bug-fix` (6 agents): triage → investigate → setup → fix → verify → PR  

**Pros:**
- **OpenClaw-native** — our platform, no adaptation needed
- YAML definitions = auditable, version-controlled workflow configuration
- SQLite state = durable, queryable, zero-infrastructure
- Cross-agent verification baked in (major quality improvement over self-reviewing agents)
- Retry + escalation = no silent failures (addresses our B-001 root cause)
- Aligns exactly with our finalized architecture (deterministic orchestration from AGENT_FINALIZED_ARCHITECTURE.md)

**Cons:**
- Node.js ≥22 required (check: Mac mini has Node v25.6.0 ✅)
- Pre-v0.6 (v0.5.1) — expect API churn in workflow YAML format
- Each agent = a separate OpenClaw session → costs accumulate for long workflows
- Not a replacement for our always-on cron loops; complementary for project-scoped work

**Risk:** LOW-MEDIUM (Node ≥22 present; YAML API may change)  
**Decision:** **ADOPT for project-scoped multi-step work.** Antfarm is the closest production-ready embodiment of our AGENT_FINALIZED_ARCHITECTURE.md. Winnie recommends Mack installs and tests on a sandboxed project task in the next cycle. Our existing cron loops remain the always-on backbone; Antfarm fills the "bounded workflow" slot.

### 1.3 Oh-My-OpenCode (OMO)

**Source:** https://github.com/code-yeongyu/oh-my-opencode (v3.4.0)

**What it is:**  
A full agent harness/plugin system for **OpenCode** (NOT OpenClaw). Provides 11 specialized agents, 41 lifecycle hooks across 7 event types, 25+ tools (LSP, AST-Grep, delegation, task management), and a Claude Code compatibility layer.

**Architecture:**
```
OpenCode → oh-my-opencode plugin → 3-tier hierarchy (Atlas → Primary Agents → Subagents) → 
  hub-and-spoke delegation → per-agent model routing (Opus/Codex/Gemini/Grok)
```

**Key features:**
| Feature | Status |
|---|---|
| Deterministic guarantees | **WEAK** — LLM-directed delegation; hub agent adapts dynamically |
| Failure handling | Lifecycle hooks can catch failures; no built-in retry mechanism |
| Context management | Subagent delegation handles context boundaries; but shared state within session |
| Install/operational burden | **HIGH** — OpenCode dependency, OAuth complications, plugin ecosystem |
| Memory model | External (Supermemory integration optional); no built-in persistence |
| Multi-agent support | ✅ Full (11 agents) |
| Target platform | **OpenCode** (NOT OpenClaw) |
| Claude OAuth | ⚠️ BLOCKED by Anthropic as of Jan 2026 |

**Key specialized agents:** Sisyphus (main), Prometheus (planner), Oracle (architecture/debugging), Librarian (docs/code search), Hephaestus (autonomy), Explorer (fast codebase grep), Multimodal Looker

**Pros:**
- Most feature-rich agent harness available (hooks, MCPs, LSP/AST tools)
- Multi-model routing natively (Opus 4.6, GPT-5.3 Codex, Gemini 3 Flash)
- Claude Code compatibility layer (bring existing workflows)
- Active development community; v3.4.0 stable

**Cons:**
- **CRITICAL: Targets OpenCode, not OpenClaw.** We run on OpenClaw. Significant adaptation required.
- **CRITICAL: Anthropic has blocked third-party OAuth access** (Jan 2026) — complicates Claude model access
- High operational burden: plugin ecosystem, OpenCode dependency, ongoing API churn
- LLM-directed delegation = less predictable than deterministic YAML workflows
- No built-in retry/escalation; failure modes are hook-dependent
- Heavy installation footprint vs our minimal-infrastructure preference

**Risk:** HIGH (platform mismatch, OAuth blocker, operational complexity)  
**Decision:** **SKIP for production.** Extract patterns only. Specifically:
  - Adopt the **agent specialization naming convention** (Oracle for architecture, Librarian for docs/search, Explorer for codebase grep) as role labels in our YAML prompts
  - Monitor their **lifecycle hook patterns** (41 hooks across 7 event types) for ideas to bring into our cron loop design
  - Do NOT install OMO; do NOT depend on OpenCode

## 3. Vector Search vs Plain SQL — Research Decision Memo

### The Question
For the planned `agent-memory.db` SQLite database: should we use `sqlite-vec` for semantic (embedding-based) vector search, or stick with plain SQL keyword/tag search?

### sqlite-vec (asg017/sqlite-vec)
**Source:** https://github.com/asg017/sqlite-vec

- Pure C extension, zero dependencies, runs anywhere SQLite runs
- Supports float, int8, binary vectors in `vec0` virtual tables
- KNN queries as standard SQL: `SELECT rowid, distance FROM vec_items WHERE sample_embedding MATCH ?`
- Faster than NumPy; "not the fastest in the world" (pre-Faiss indexed approaches)
- **Pre-v1 status** — breaking changes likely; API not stable
- Install: `pip install sqlite-vec` (Python) or `npm install sqlite-vec` (Node.js)
- Alternative: `sqliteai/sqlite-vector` (BLOB-based, SIMD-accelerated, no virtual tables needed)

### Plain SQL Keyword/Tag Search
- Zero extension dependency; works with vanilla SQLite
- FTS5 full-text search built into SQLite ≥3.35 — covers substring/word matching
- Does NOT handle semantic similarity ("find memories about deployment that don't use the word 'deploy'")
- Adequate if memories are well-tagged and retrieval is always exact/keyword based

### Recommendation
| Approach | When to use |
|---|---|
| Plain SQL + FTS5 | **Phase 1** — bootstrap agent memory; get something running |
| sqlite-vec + embeddings | **Phase 2** — once dataset >500 memories; enable semantic retrieval for Winnie/Rosie context loading |

**Decision:** Start with **plain SQL + FTS5** for Phase 1 (zero extension risk). Mack installs and sandboxes `sqlite-vec` in parallel. Winnie will run concrete performance tests in the next cycle (after `agent-memory.db` schema exists).
