# Mack — memU Resilience + SQLite Hardening

**Date:** 2026-02-23 00:17 EST  
**Task:** MEMU-RESILIENCE-HARDENING  
**Version:** 1.5.0 → 2.0.0  

## Summary

Audited and hardened `memu_server/server.py` for atomic persistence, idempotency, TTL/GC, crash recovery, and WAL/replay safety. All 8 changes are backward-compatible with existing data.

## Issues Found & Fixed

### 1. Connection Leak (Critical)
**Before:** `_conn()` created a new `sqlite3.connect()` on every call. `_insert_entry()` opened its own connection separate from `store()`, breaking transactional atomicity.  
**After:** Thread-local connection pool (`threading.local`). Each thread reuses one connection. `_insert_entry_raw()` accepts a connection argument — callers control the transaction scope.

### 2. Split-Connection Atomicity (Critical)
**Before:** `store()` used one connection for the idempotency check, then `_insert_entry()` used a *different* connection for the insert. Race window: two concurrent stores with the same idempotency key could both pass the check before either inserts.  
**After:** Single connection for check → insert → commit. Thread lock still provides serialization, but the connection reuse is the real fix.

### 3. NULL Idempotency Keys (High)
**Before:** 339 of 372 entries had `NULL` idempotency_key — any entry stored without an explicit key had no dedup protection.  
**After:** Content-hash fallback: `ch-sha256(agent_id:key:content)[:24]` generates a deterministic key when none is provided. Identical payloads produce the same hash → INSERT OR IGNORE catches the duplicate.

### 4. WAL Bloat (Medium)
**Before:** WAL file was 1.1MB and growing. No auto-checkpoint. Only `wal_autocheckpoint` default (1000 pages) was active, but no periodic explicit checkpoint.  
**After:** `PRAGMA wal_autocheckpoint=1000` plus explicit `PRAGMA wal_checkpoint(PASSIVE)` every 5 minutes. Final `TRUNCATE` checkpoint on shutdown via `atexit`.

### 5. GC Only at Startup (Medium)
**Before:** `_collect_garbage()` ran once at init. A server running for days would never GC.  
**After:** GC runs after every write if >6 hours since last run (`_periodic_maintenance()`).

### 6. No Crash Recovery (High)
**Before:** Server opened DB and assumed integrity. A crash mid-WAL-replay could leave the DB inconsistent.  
**After:** `_startup_integrity_check()` runs `PRAGMA integrity_check` at startup. If it fails, attempts `PRAGMA wal_checkpoint(TRUNCATE)` recovery. Logs all outcomes.

### 7. Event Log Not Crash-Safe (Medium)
**Before:** `log_event` used Python `open("a") + f.write()` — no fsync, buffered I/O. A crash between write and OS flush loses the event.  
**After:** Uses `os.open(O_WRONLY|O_APPEND|O_CREAT)` + `os.write()` + `os.fsync()`. Atomic at OS level for single-line appends.

### 8. No Graceful Shutdown (Low)
**Before:** `Ctrl-C` killed the server with no cleanup. Pending WAL could be large.  
**After:** `atexit.register(self._shutdown)` runs `PRAGMA wal_checkpoint(TRUNCATE)` to flush WAL to main DB.

## Additional Improvements
