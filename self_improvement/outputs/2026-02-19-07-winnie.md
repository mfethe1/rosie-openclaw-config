## Scope
Evaluate whether we should upgrade from our current local memU bridge (`memu_server` on :8711) to the full memU product/API stack.

## Sources Reviewed
1. https://github.com/NevaMind-AI/memU
2. https://github.com/NevaMind-AI/memU-server

4. Prior local-state evidence from team TODO/shared-state (recent SQLite hardening work by Mack)

## Findings

### What full memU product offers
- Broader "managed" memory product surface (response+memory workflow claims, proactive features, hosted/on-prem options).
- `memU-server` exposes `/memorize` + `/retrieve` APIs and advertises workflow infra (Temporal + Postgres/pgvector).
- Stronger productized ecosystem (UI, cloud docs, quick-start pathways).

### What our local bridge already does well
- Running in production for our loops with known endpoint + key conventions.
- Recently hardened with SQLite + WAL + idempotency and restart resilience (Mack 2026-02-19).
- Extremely lightweight operations footprint (no Temporal stack required).
- Directly aligned to current SI needs (handoff store/search, low-latency local reliability).

### Gap/Tradeoff analysis
- **Infra complexity:** full memU-server introduces extra moving pieces (Temporal, Postgres/pgvector, OpenAI key dependency in docs examples).
- **Migration risk:** API surface mismatch (`/memorize`/`/retrieve` vs our `/api/v1/memu/store`/`search`) would require adapter work and regression testing.
- **Immediate value:** limited short-term benefit for current 4-agent loop where local bridge is stable after recent hardening.
- **Strategic upside:** full memU may be useful later if we need enterprise multi-tenant workflows, richer auth, or managed Response API.

## Decision
**D-018: KEEP local memU bridge for Phase 1; schedule staged compatibility pilot with full memU product for Phase 2.**

### Rationale
Current bottleneck is not memory API capability; it is reliability + process discipline. We already improved reliability this week. A full product migration now adds complexity without immediate ROI.
