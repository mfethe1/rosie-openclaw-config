## 2. Antfarm (snarktank/antfarm)

### What's new
- SQLite-backed pipeline state at `~/.openclaw/antfarm/antfarm.db` — durable run tracking across cron cycles.
- 5-agent pipeline: planner → developer → verifier → tester → reviewer. Role separation is strong.
- Predefined workflows: feature-dev, bug-fix, security-audit.
- Autonomous cron execution is the core value prop — fire and forget with DB-level audit trail.
- Cost risk flagged: unmonitored antfarm pipelines reported at $100+/month typical, $3,600/month extreme case.

### Evaluation matrix

| Dimension | Score | Notes |
|---|---|---|
| Deterministic execution | HIGH | Explicit step ordering in SQLite; each stage is a discrete agent handoff |
| Verification gates | HIGH | Verifier + tester are dedicated pipeline stages — structural enforcement |
| Cost/maintenance burden | HIGH RISK | No native cost caps; SQLite DB grows unbounded; runaway pipeline risk without budget controls |
| Model portability | MEDIUM | OpenClaw-native; depends on OpenClaw model routing config |

### Recommendation: **TEST** (controlled pilot, budget-gated)
- Antfarm's verifier/tester stage separation is the strongest gate model of the three tools.
- Block: no cost cap enforcement. Do NOT enable autonomous cron without a spend budget trigger.
- Next: run antfarm feature-dev pipeline on one low-stakes task (docs generation) with a manual cost checkpoint after each stage. Evaluate whether the gate quality justifies the additional agent overhead vs our current OMO flow.

## 3. Ralph Loop (iannuttall/ralph + openclaw-skills-ralph-loop-agent)

### What's new
- OpenClaw skill published 4 days ago (lobehub.com): backpressure, sandboxing, explicit completion conditions.
- pageai.pro writeup (3 weeks ago) confirms the canonical model: one task per loop iteration, LLM decides priority.
- DEV.to article (3 days ago) directly compares Ralph vs Antfarm: confirms **no plugin/tool ecosystem** — tool access must be hardcoded per loop.
- stacktoheap.com implementation notes: "deterministically allocate the stack" per iteration.
- KenHuang Substack: RALPH is a runtime execution harness, not a ReAct-style reasoning loop. Distinct paradigm.

### Evaluation matrix

| Dimension | Score | Notes |
|---|---|---|
| Deterministic execution | HIGH (per-iteration) | Single task per loop + explicit stack allocation = reproducible iteration behavior |
| Verification gates | LOW | No built-in verification layer. Gates must be externally scripted. |
| Cost/maintenance burden | LOW | Minimal infrastructure — no DB, no cron state, no plugin system |
| Model portability | HIGH | Model-agnostic by design; works with any provider |

### Recommendation: **TEST** (narrow use case — reproducible script generation)
- Ralph excels for isolated, repeatable single-task execution where verification is NOT needed inside the loop.
- Weakness is our biggest need: no verification gates, no tool ecosystem per agent.
- Next: evaluate Ralph for one specific use case — automated script/boilerplate generation where output is deterministic and testable externally (e.g., Lenny runs a smoke test post-loop).

## Cross-cutting signals

1. **Cost discipline is the common failure mode.** All three tools can generate runaway spend. OMO + Antfarm most at risk due to cron+multi-agent. Budget gates are mandatory before any new autonomous pipeline.
2. **Verification gate quality**: Antfarm > OMO > Ralph. If we want structural QA enforcement, Antfarm's pipeline stages are the strongest architecture.
3. **Model portability**: OMO = Ralph > Antfarm (Antfarm is tightly OpenClaw-coupled).
4. **No tool has deterministic end-to-end execution.** All rely on LLM decisions at some layer. "Deterministic" means iteration structure is fixed, not that outputs are reproducible.
