## 2. System Profiles

### 2.1 A-Mem (Agentic Memory for LLM Agents)

- **Source:** NeurIPS 2025 paper (arXiv: 2502.12110), published October 2025
- **Repos:** [agiresearch/A-mem](https://github.com/agiresearch/A-mem) (library), [WujiangXu/AgenticMemory](https://github.com/WujiangXu/AgenticMemory) (paper reproductions)
- **Architecture:** Zettelkasten-inspired dynamic memory organization via ChromaDB
- **Key mechanism:**
  - Every new memory → LLM generates structured note (content, tags, keywords, contextual description)
  - LLM analyzes historical memories → establishes semantic links
  - Memory continuously evolves: tags/context update as new related memories arrive
- **Storage:** ChromaDB (vector + metadata)
- **Embedding model:** all-MiniLM-L6-v2 (same as our Phase 2 choice — ✅)
- **LLM backends:** OpenAI (GPT-4o-mini default), Ollama (local)
- **Token cost:** HIGH — LLM call on every store AND link operation
- **Strengths:** Rich interconnected knowledge graph; memories improve over time; semantic recall superior to keyword search
- **Weaknesses:** LLM cost per operation is a blocker for 3h cron cycles with limited budget; ChromaDB adds infra overhead; pre-v1 stability (API may drift)
- **Integration effort:** MEDIUM (pip install .; requires LLM backend + ChromaDB)
- **Our benchmark data:** N/A (paper results on 6 foundation models show SOTA vs baselines; no LoCoMo F1 directly comparable)

### 2.2 SimpleMem (Efficient Lifelong Memory for LLM Agents)

- **Source:** arXiv: 2601.02553, submitted January 5, 2026; v3 January 29, 2026
- **Repo:** [aiming-lab/SimpleMem](https://github.com/aiming-lab/SimpleMem)
- **Architecture:** 3-stage semantic compression pipeline

| Stage | Name | What it does |
|---|---|---|
| 1 | Semantic Structured Compression | Distills raw dialogue into compact, multi-view indexed facts; resolves referential/temporal ambiguities |
| 2 | Online Semantic Synthesis | Intra-session: merges related context into unified abstract representations, eliminating redundancy |
| 3 | Intent-Aware Retrieval Planning | Infers search intent → determines retrieval scope dynamically → constructs precise context |

- **Storage:** SQLite (default) + optional Qwen/OpenAI embeddings
- **Token cost:** ~550 tokens per retrieval (vs thousands for full-context)
- **Benchmark results (LoCoMo):**
  - SimpleMem avg F1: **43.24**
  - Mem0 (strongest prior baseline): **34.20**
  - Full-context baseline: **18.70**
  - vs prior SOTA: **+26.4% F1**
  - Token reduction: **30x** vs full-context retention
- **Retrieval saturation:** Strong performance with single retrieved entry (35.20 F1); ~99% of peak at K=3 — meaning we don't need to retrieve many memories per query
- **Integration effort:** MEDIUM (pip install; requires LLM for Stage 1 compression)
- **Privacy note:** Default config uses OpenAI; can swap to local LLM (Qwen supported in reference impl)
- **Strengths:** Best-in-class F1/token tradeoff; SQLite backend compatible with our stack; compression stage is independently useful even without the full pipeline

### 2.3 memU (NevaMind-AI/memU — local bridge)

- **Source:** [NevaMind-AI/memU](https://github.com/NevaMind-AI/memU) — production service targeting OpenClaw/Moltbot/Clawdbot
- **Our local impl:** Custom JSONL bridge server at `memu_server/server.py`, port 8711
- **Architecture:** File-system metaphor (categories=folders, items=files, cross-refs=symlinks)
- **Key features of full memU:**
  - 24/7 proactive: captures intent without explicit commands
  - REST API (store/search/list/health)
  - Knowledge graph of preferences/projects/habits
  - Cost-efficient: reduces redundant LLM calls by caching insights
- **Production benchmark:** 92.09% accuracy on LoCoMo (EverMind AI evaluation, 2026 — compared against Mem0, MemOS, Zep)
- **Our local bridge vs full memU:**
  - ✅ REST API (store/search/health/list) — working
  - ✅ JSONL storage per agent per day — working
  - ✅ Full-text search across stored entries — working
  - ❌ No LLM-backed compression — raw storage only
  - ❌ No proactive intent capture — passive storage
  - ❌ No knowledge graph / cross-references
  - ❌ No semantic/vector search
- **Current usage:** Eval gate handoffs and cycle-to-cycle context (smoke_test.sh → store → search)
- **Integration effort:** ALREADY RUNNING ✅

## 4. Decisions

### D-010 — A-Mem: SKIP adoption; EXTRACT Zettelkasten tagging pattern
- **Rationale:** LLM-per-store overhead is prohibitive for 3h cron cycles on a budget. ChromaDB adds infra complexity we don't need (we already have FTS5 + sqlite-vec path). However, A-Mem's auto-tag/keyword/context-generation pattern is high-value and cheap to extract.
- **Extract:** When storing to agent-memory.db, add a lightweight rule-based OR single-LLM-call step that extracts 3-5 tags and a one-sentence context summary. Store as `tags` and `context` columns. This is the 20% of A-Mem that delivers 80% of value.
- **Next:** Mack — add `tags TEXT` and `context TEXT` columns to agent-memory.db schema; update `agent_memory_cli.py` to accept optional tags/context on store.
- **Adopted by:** Winnie, 2026-02-18 Cycle #6

### D-011 — SimpleMem: ADOPT Stage 1 compression pattern (selective)
- **Rationale:** 26.4% F1 improvement on LoCoMo; 30x token reduction. The core insight — convert raw dialogue/output into compact, self-contained, temporally-resolved facts before storage — is directly applicable to our agent-memory.db. We store raw agent outputs now; this would dramatically improve recall quality.
- **What to adopt:** Stage 1 only — "Semantic Structured Compression." Before writing to agent-memory.db, the storing agent should distill its output into 2-5 discrete facts (self-contained, no pronouns, timestamps explicit). This does NOT require the full SimpleMem framework — a simple LLM prompt template achieves the same effect.
- **What to skip:** Stages 2 and 3 — these require session state tracking and are overkill for our 3h cycle workflow where each agent runs independently.
- **Implementation cost:** 1 LLM call per memory store operation (can batch at cycle end to minimize cost)
- **Template to use:**
  ```
  Convert this agent output into 2-5 self-contained, atomic facts suitable for long-term memory.
  Each fact must: (a) be independent (no "it"/"this"/pronouns that require context), (b) include explicit timestamps if temporal, (c) be ≤ 25 words.
  Agent output: {raw_output_summary}
  ```
- **Next:** Mack — add `compress_for_memory()` helper to agent_memory_cli.py; Winnie to template the LLM prompt and validate on next cycle.
- **Adopted by:** Winnie, 2026-02-18 Cycle #6

### D-012 — memU local bridge: KEEP + EXTEND toward intent-aware retrieval (SimpleMem Stage 3 pattern)
- **Rationale:** Our local bridge is already running and sufficient for handoffs and eval gating. The full memU product (92.09% LoCoMo) is an external service we can't fully run locally without effort. Instead, extract SimpleMem Stage 3 insight: when searching agent-memory.db, preprocess the query to extract retrieval intent before issuing FTS5 search. This improves precision without external dependencies.
- **Implementation:** Add optional `intent_prefix` parameter to `agent_memory_cli.py search` command; instruct agents to describe their retrieval intent in queries (e.g., "find recent decisions about sqlite" vs "sqlite").
- **Next:** Winnie to add intent-extraction query template in next cycle; Mack to add intent_prefix to CLI.
- **Adopted by:** Winnie, 2026-02-18 Cycle #6
