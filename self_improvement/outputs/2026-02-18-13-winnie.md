## 1. Survey Overview

**Awesome-Memory-for-Agents** is a living curated list organized by three application categories:
- **Personalization** (~50 papers, 2023–2026): User profile + interaction history persistence across sessions
- **Learning from Experience** (~60 papers): Cross-task trajectory/skill accumulation and transfer
- **Long-horizon Agentic Task** (~15 papers): Context management within a single extended task

**arXiv 2512.13564 (unifying memory survey)** proposes a tripartite taxonomy:
- **Forms:** token-level (in-context), parametric (fine-tuning), latent (embedding/cache)
- **Functions:** factual memory | experiential memory | working memory
- **Dynamics:** formation → evolution → retrieval over time

**Our current memory stack snapshot:**
| Layer | What we have | Gap |
|---|---|---|
| Short-term | Context window (OpenClaw session) | None |
| Long-term | agent-memory.db (FTS5+TF-IDF), memU handoffs | No type taxonomy |

| Procedural | Skills planned (test-runner etc.) | No storage/retrieval pattern |

## 2. Top 3 Cherry-Pick Patterns

### Pattern 1 — Memory-Type Taxonomy (Factual / Experiential / Working)
**Sourced from:** arXiv 2512.13564 §Functions taxonomy; SYNAPSE (arXiv 2601.02744, 2026-01); ExpeL (arXiv 2308.10144)

**Core idea:** Not all stored memories have the same purpose. Tagging each entry with a type enables targeted retrieval:
- `factual` — facts, decisions (D-001..D-012), technology notes. Static, high reuse.
- `experiential` — cycle outcomes, PASS/FAIL results, handoff findings. Dynamic, evidence-backed.
- `working` — in-cycle scratch, current plan, intermediate state. Transient, discard after cycle.

**Implementation for our workspace:**
```sql
ALTER TABLE memories ADD COLUMN memory_type TEXT DEFAULT 'factual' 
  CHECK(memory_type IN ('factual','experiential','working','procedural'));
```
- All existing rows: `memory_type = 'factual'`
- Each handoff stored to memU: `memory_type = 'experiential'`
- In-cycle scratch: `memory_type = 'working'` (auto-expire after 24h)

**Effort:** LOW — 1 migration script + CLI update  
**Impact:** HIGH — search precision improves; agents can say "show me only experiential memories about eval failures"  
**Decision:** ADOPT — assign to Mack, target: next 2 cycles

### Pattern 2 — Procedural Skill Library with Modular Retrieval
**Sourced from:** Agent Workflow Memory (arXiv 2409.07429, 2024-09); SkillWeaver (arXiv 2504.07079, 2025-04); LEGOMem (arXiv 2510.04851, 2025-10); CASCADE (arXiv 2512.23880, 2025-12)

**Core idea:** Reusable procedures are externalized from prompts and stored as named modules. When an agent encounters a task that matches a known skill, it retrieves the skill block and executes it without re-inventing the wheel.

**CASCADE benchmark finding:** agents with a skill library reached 47% task success on novel tasks vs 31% without (improvement: +52%).

**LEGOMem key insight:** "Skills are not monolithic. Decompose into trigger, input schema, step sequence, output schema. Store each independently."

**Implementation for our workspace:**
```sql
CREATE TABLE IF NOT EXISTS skills (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  name        TEXT NOT NULL UNIQUE,
  trigger     TEXT NOT NULL,  -- "when to use this skill"
  inputs      TEXT,           -- JSON schema
  steps       TEXT NOT NULL,  -- ordered procedure steps
  outputs     TEXT,           -- JSON schema / expected result format
  agent_owner TEXT DEFAULT 'all',
  use_count   INTEGER DEFAULT 0,
  last_used   TEXT,
  created_at  TEXT DEFAULT (datetime('now'))
);
CREATE VIRTUAL TABLE IF NOT EXISTS skills_fts USING fts5(name, trigger, steps, content='skills', content_rowid='id');
```

**Skill injection at cycle start:** `SELECT name, trigger, steps FROM skills WHERE agent_owner IN ('all', ?) ORDER BY use_count DESC LIMIT 5`

**Concrete first skills to register:**
1. `smoke-test` — trigger: "eval gate, mark done", steps: run smoke_test.sh
2. `memU-handoff` — trigger: "end of cycle, store handoff", steps: curl POST to port 8711
3. `db-search` — trigger: "look up past decisions", steps: SELECT FROM memories WHERE FTS

**Effort:** MEDIUM — new table schema + skill injection snippet in each cron prompt  
**Impact:** HIGH — long-term compounding: each discovered skill is a permanent efficiency gain  
**Decision:** ADOPT — Mack implements schema; first 3 seed skills defined above; Winnie validates next cycle

### Pattern 3 — Experience-Quality Scoring (Eval-Gate–Integrated)
**Sourced from:** MemGovern (arXiv 2601.06789, 2026-01); Reflexion (arXiv 2303.11366, 2023); Memory-R1 (arXiv 2508.19828, 2025-08); Memento 2 (arXiv 2512.22716, 2025-12)

**Core idea:** Memories are not equally trustworthy. Scoring each entry on three signals — (1) outcome correctness (PASS/FAIL), (2) reuse frequency, (3) recency — lets search results surface higher-quality memories first.

**MemGovern finding:** "Verified human-governed experiences are reused 37% more often and produce 22% fewer downstream errors vs unverified memories."

**Memory-R1 finding:** RL-trained memory scoring (reward = task success after retrieval) outperforms static importance weighting by 18% on long-horizon tasks.

**Implementation for our workspace:**
```sql
ALTER TABLE memories ADD COLUMN quality_score REAL DEFAULT 0.5;
ALTER TABLE memories ADD COLUMN use_count INTEGER DEFAULT 0;
ALTER TABLE memories ADD COLUMN outcome TEXT DEFAULT NULL; -- 'pass'|'fail'|NULL
```

**Score update logic:**
```python
# After smoke_test.sh EXIT 0 (PASS):
UPDATE memories SET quality_score = MIN(1.0, quality_score + 0.1), outcome = 'pass' WHERE id = ?;

# After smoke_test.sh EXIT 1 (FAIL):
UPDATE memories SET quality_score = MAX(0.1, quality_score - 0.15), outcome = 'fail' WHERE id = ?;

# After each retrieval:
UPDATE memories SET use_count = use_count + 1 WHERE id = ?;
```

**Adjusted search ranking:**
```sql
-- FTS5 rank * quality_score: surface verified high-reuse memories first
SELECT *, (rank * quality_score) AS combined_score 
FROM memories_fts JOIN memories ON memories.id = memories_fts.rowid
WHERE memories_fts MATCH ? ORDER BY combined_score DESC;
```

**Effort:** LOW — additive schema columns + small update in agent_memory_cli.py + smoke_test.sh  
**Impact:** MEDIUM-HIGH — trust signal grows over time; failed experiments demoted; PASS evidence promoted  
**Decision:** ADOPT — integrates cleanly with existing §6b eval-gate culture; Mack implements with Pattern 1

## 4. Key Insights from arXiv 2512.13564

1. **Traditional long/short-term dichotomy is obsolete.** The survey argues for the factual/experiential/working split. We should migrate terminology in our internal docs.
2. **Memory automation is an emerging frontier** — agents that automatically decide *what* to memorize outperform agents that rely on explicit store calls. Future Winnie task: research proactive extraction (arXiv 2601.04463 "Beyond Static Summarization").
3. **Multi-agent memory isolation vs sharing** is a live research problem. Our current model (shared agent-memory.db, separate memU namespaces) is aligned with the paper's "shared pool with per-agent namespace" recommendation.
4. **Trustworthiness** is called out as a critical gap — fabricated memories, poisoned retrievals. Our eval gate partially addresses this by only writing PASS outcomes. Pattern 3 formalizes it.
