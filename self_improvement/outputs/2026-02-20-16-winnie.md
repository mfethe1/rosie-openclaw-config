## Paper 1: MAGMA (arXiv:2601.03236, Jan 6 2026)
**Full title:** "MAGMA: A Multi-Graph based Agentic Memory Architecture for AI Agents"
**Authors:** Dongming Jiang, Yi Li, Guanpeng Li, Bingzhe Li (UT Dallas / U Florida)
**Benchmarks:** LoCoMo + LongMemEval — outperforms SOTA agentic memory systems

### Core Problem
Existing MAG systems use **semantic similarity over monolithic stores**, entangling temporal, causal,
and entity information in one flat index. Queries like "what caused X?" incorrectly compete with
"what relates to X?" — leading to poor alignment between intent and retrieved evidence.

### Architecture
Each memory item stored across **4 orthogonal graphs**:
| Graph | Models | Query use case |
|---|---|---|
| **Semantic** | Topic/concept proximity | "what relates to X?" |
| **Temporal** | Time-ordered sequence | "what happened before/after Y?" |
| **Causal** | Cause-effect links | "what fixed/triggered Z?" |
| **Entity** | Entity co-occurrence | "all memories involving agent A?" |

Retrieval = **policy-guided graph traversal**, not cosine similarity.

### Gap vs Our System
| MAGMA concept | Our system | Status |
|---|---|---|
| 4 orthogonal graphs | Single FTS5 flat index | ❌ Missing separation |
| Query-type routing | All queries same path | ❌ No routing layer |
| Transparent reasoning paths | No trace | ❌ |

### Decision D-024: SKIP full MAGMA; EXTRACT query-type routing
**Verdict:** SKIP (4-graph infra = significant complexity for 4-agent scale).
**EXTRACT** lightweight query-type dispatch on existing FTS5+TF-IDF stack:

```
--query-type temporal  → ORDER BY stored_at DESC, optional date window
--query-type causal    → body CONTAINS "caused|fixed|resolved|triggered|led to"
--query-type entity    → tags OR context CONTAINS query term
--query-type factual   → default FTS5 rank (unchanged)
```

**Owner:** Mack | **Effort:** LOW (4 SQL paths + CLI flag)

## Paper 2: EverMemOS (arXiv:2601.02163, Jan 5 2026)
**Full title:** "EverMemOS: A Self-Organizing Memory OS for Structured Long-Horizon Reasoning"
**Authors:** Chuanrui Hu et al. (code available)
**Benchmarks:** LoCoMo + LongMemEval — **state-of-the-art** + PersonaMem v2 profiling

### Architecture: Engram-Inspired 3-Phase Lifecycle

```
Dialogue stream
       ↓
[1] Episodic Trace Formation
    → MemCells: episode narrative + atomic facts + Foresight signals (time-bounded plans/goals)
       ↓
[2] Semantic Consolidation
    → MemCells clustered → MemScenes (thematic groups)
    → User profile evolution (stable vs transient states)
       ↓
[3] Reconstructive Recollection
    → MemScene-guided retrieval (scene first, then cell)
    → Foresight validity check (expired signals filtered)
    → Compose necessary+sufficient context
```

### Key Innovation 1: Foresight Signals
Time-bounded expectations stored at formation: "Expect to need X in the next 3 hours."
- Carry a validity interval — retrieved only while valid
- Independently confirms our **D-021 Working Memory TTL** decision is correct

### Key Innovation 2: MemScene → MemCell 2-Stage Retrieval
Retrieve at **scene level first** (fast, tags-based) then drill to cells (precise, FTS5).
Improves efficiency — avoids full FTS5 scan when a coarse topic match suffices.

### Gap vs Our System
| EverMemOS concept | Our system | Status |
|---|---|---|
| MemCell (episode + facts + foresight) | knowledge_extractor.py entries | ✅ Close match |
| Foresight signals with TTL | D-021 working memory TTL | ✅ D-021 covers this |
| MemScene thematic clusters | D-022 consolidator (planned) | ✅ D-022 covers this |
| 2-stage scene-then-cell retrieval | Single-stage FTS5 | ❌ Gap |
| User profile evolution | Not present | ❌ Out of scope |

### Decision D-025: SKIP full EverMemOS; EXTRACT Foresight practice + 2-stage retrieval

**D-025a: Foresight-as-Practice (VERY LOW effort)**
Each agent writes 1-2 foresight working-memory items at cycle end:
```bash
python3 agent_memory_cli.py store \
  --agent winnie --type working --expires-at "+3h" \
  --topic "foresight:next-cycle" \
  --tags "foresight,working" \
  --body "Validate D-023 reflect hook wired in smoke_test.sh next cycle"
```
Formalises what handoff notes do informally — makes it machine-readable for STEP 1.

**D-025b: 2-Stage Search (LOW effort)**
```
Stage 1: tags OR context LIKE '%query%'   (fast coarse match — "scene level")
Stage 2: FTS5 full-text match on body     (precise — "cell level")
Merge, deduplicate, rank: provenance_score * quality_score
```
