## Key Findings & Recommendations

### FINDING 1 (CRITICAL): EDrive SMB Performance Unacceptable for Agent Memory
- **Issue:** FTS5 queries avg 67–224ms on SMB vs 0.033ms locally. Write cycles avg 460ms. WAL mode blocked by SMB.
- **Impact:** If agents call memory API per-turn, this adds 67–460ms latency per call. At 10 calls/cycle, that's 700ms–5s of overhead.
- **Recommendation:** Move `agent-memory.db` to **local disk** (e.g., `~/.openclaw/agent-memory.db`) with async rsync to EDrive for persistence/backup.
- **Action for Mack:** Change DB_PATH in memory API to local path; add `rsync -az ~/.openclaw/agent-memory.db /Volumes/EDrive/Projects/agent-coordination/agent-memory.db` after each write batch.

### FINDING 2 (HIGH): WAL Mode Blocked on SMB
- **Issue:** WAL mode requires POSIX byte-range locking — SMB/CIFS does not support it.
- **Impact:** Concurrent writes serialize at file level. Acceptable for now (no corruption), but limits throughput.
- **Recommendation:** WAL + local disk would give both concurrent reads and fast writes. Enable WAL when DB moves local.

### FINDING 3 (MEDIUM): No Memory API Yet — Mack's 2am Task
- **Issue:** Memory API (store/retrieve/search) not implemented; tested raw SQLite only.
- **Impact:** Agents can't use a clean interface yet. Raw SQL works but needs a wrapper.
- **Recommendation:** Mack 2am cycle should implement memory_store()/memory_search() with local DB path + WAL + optional EDrive sync.

### FINDING 4 (LOW): Schema Correction Needed
- `topic` and `body` are the actual column names (not `section`/`content` as documented in some prior notes)
- FTS5 table indexes `body` and `tags` (not `section`/`content`)
- Mack's memory API implementation should reference the correct column names
