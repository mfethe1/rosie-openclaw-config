## 1) oh-my-opencode

### Axis assessment
- **Deterministic execution guarantees:** **MEDIUM-HIGH**
  - Strong primitives (hash-anchored edits, intent gate, task routing), but runtime still depends on orchestration policy quality and prompt discipline.
- **Verification gates:** **MEDIUM**
  - LSP/AST/diagnostic tooling is strong, but hard fail-closed gates are not universally mandatory in all usage paths.
- **Cost/maintenance burden:** **HIGH**
  - High-velocity repo + large issue queue increases integration churn and ongoing maintenance load.
- **Model portability:** **HIGH (design), MEDIUM-HIGH (operations)**
  - Multi-model strategy is a clear advantage; practical portability still requires active config hygiene and fallback testing.

### Recommendation
- **KEEP:** multi-model category routing + provider fallback chain design.
- **STOP:** assuming defaults are stable across upgrades (explicitly pin operational assumptions).
- **TEST:** controlled provider-failover drill with verification invariants preserved (same gates regardless of provider swap).

## 2) Antfarm

### Axis assessment
- **Deterministic execution guarantees:** **HIGH (design), MEDIUM-HIGH (runtime)**
  - YAML step ordering and isolated stages are deterministic by design; runtime reliability still constrained by unresolved edge-case backlog.
- **Verification gates:** **HIGH**
  - Verify/review roles are first-class and explicit in workflow definitions.
- **Cost/maintenance burden:** **MEDIUM-HIGH**
  - Lightweight architecture (YAML + SQLite + cron) is cost-efficient, but maintenance risk remains due to backlog and single-maintainer concentration.
- **Model portability:** **MEDIUM-HIGH**
  - Broad portability inside OpenClaw ecosystem; less portable outside that runtime.

### Recommendation
- **KEEP:** explicit staged verification contract (implementer ≠ verifier).
- **STOP:** waiting for upstream reliability protections (especially circuit-breaker behavior).
- **TEST:** per-repo lock + idempotent resume on one cron-triggered workflow to prove overlap prevention and spend control.

## 3) Ralph-loop style tooling

### Axis assessment
- **Deterministic execution guarantees:** **HIGH for single-loop pattern**
  - Fresh-context-per-iteration + persistent progress file remains one of the most robust anti-context-rot patterns.
- **Verification gates:** **MEDIUM-HIGH**
  - Strong when test/typecheck gates are mandatory; weak if teams run Ralph without hard checks.
- **Cost/maintenance burden:** **LOW-MEDIUM**
  - Mechanically simple, but can run expensive without explicit budget/circuit-breaker constraints.
- **Model portability:** **HIGH**
  - Pattern-level portability is excellent; works across tools/providers.

### Recommendation
- **KEEP:** fresh-context iteration loop + persistent progress artifact.
- **STOP:** multi-agent loop complexity for deterministic cron paths (use only where parallelism benefit is proven).
- **TEST:** dual-condition loop exit (completion marker + explicit exit signal) with hourly budget cap.
