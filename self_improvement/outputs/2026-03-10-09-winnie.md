# Competitor Sweep: Agent Orchestration Tooling
**Date:** 2026-03-10 09:00 AM
**Target:** oh-my-opencode, antfarm, Ralph-loop

## 1. oh-my-opencode (oh-my-openagent)
* **Overview:** Multi-model agent orchestration harness for OpenCode featuring batteries-included agents (Sisyphus, Librarian, Explore, Oracle).
* **Deterministic Execution:** Low/Medium. Relies on prompt-based coordination rather than strict programmatic state machines.
* **Verification Gates:** Relies on Oracle/peer-review agent rather than code-driven testing.
* **Cost/Maintenance Burden:** Medium to High overhead due to orchestrating multiple autonomous agents simultaneously.
* **Model Portability:** Excellent (75+ models supported).
* **Recommendation:** **Keep** the specialized agent role abstractions. **Stop** relying on unstructured agent-to-agent chatter. **Test** their new MCP hooks for integration.

## 2. Antfarm
* **Overview:** One-command agent team builder in OpenClaw (planner, developer, verifier, tester, reviewer).
* **Deterministic Execution:** Medium. Workflows are repeatable with run records per task.
* **Verification Gates:** Stronger separation of concerns with dedicated verifier and tester agents.
* **Cost/Maintenance Burden:** Low setup ("zero infrastructure"), but multi-agent token usage per task is high.
* **Model Portability:** Tied to OpenClaw ecosystem configurations.
* **Recommendation:** **Keep** the native OpenClaw integration and run-record philosophy. **Test** the verifier/tester isolated workflow.

## 3. Ralph-loop
* **Overview:** Iterative pattern wrapping agent generation in an outer loop until a `verifyCompletion` function confirms success.
* **Deterministic Execution:** Very High. Forces continuous iteration until success or safety limit is reached.
* **Verification Gates:** Strict programmatic verification (e.g., test passes, linter clear).
* **Cost/Maintenance Burden:** High risk of token looping costs if verification criteria are impossible; low architectural maintenance.
* **Model Portability:** High (integrates via standard SDKs or standalone CLI).
* **Recommendation:** **Test** implementing a Ralph-loop style verification gate for our core task execution to guarantee completion before handoff. **Stop** assuming open-loop generation is complete without a programmatic gate.
