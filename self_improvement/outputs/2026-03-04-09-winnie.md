# Competitor Sweep — 2026-03-04 09:00 (Winnie)

Scope: oh-my-opencode, Antfarm, Ralph-loop style tooling.
Decision lens: deterministic execution guarantees, verification gates, cost/maintenance burden, model portability.

## 1) oh-my-opencode

### New signal this cycle
- New post-v3.9.0 release activity on Mar 2 focused on security + determinism hardening:
  - HTTP hook URL scheme validation (http/https only)
  - Deterministic/unique hook message-part IDs (process-prefixed)
  - Monotonic ordering fixes for context injection
  - Hashline edit flipped to opt-in by default (safer default, migration cost)

### Axis assessment
- **Deterministic execution:** **UP** (monotonic ordering + ID collision fixes materially reduce restart/parallel nondeterminism)
- **Verification gates:** **MEDIUM** (tooling exists, but still largely policy/enforcement dependent)
- **Cost/maintenance burden:** **MEDIUM-HIGH** (high release velocity + behavior-changing defaults like hashline opt-in)
- **Model portability:** **MEDIUM** (multi-provider pressure remains; legacy model-drift issues still show up in issue backlog)

### Recommendation
- **KEEP:** deterministic ID + monotonic ordering patterns as reference architecture.
- **STOP:** assuming hashline-edit availability as implicit default in any automation path.
- **TEST:** one internal hook-injection path with explicit URL-scheme allowlist + deterministic ID prefix strategy.

## 3) Ralph-loop style tooling

### New signal this cycle
- snarktank/ralph: no formal releases, but active PRs (including proposals for multi-agent parallelization).
- Ralph ecosystem split:
  - conservative loop pattern (fresh context + persisted progress)
  - feature-rich forks (e.g., dual-condition exit gate, hourly rate limiting, stronger circuit-breakers).
- Geoffrey Huntley’s current writing continues to favor monolithic single-loop execution over multi-agent complexity for determinism.

### Axis assessment
- **Deterministic execution:** **HIGH when monolithic single-task loop; LOWER when multi-agent forks are layered in**
- **Verification gates:** **MEDIUM-HIGH** (strong only when tests/typecheck/explicit gates are non-optional)
- **Cost/maintenance burden:** **LOW-MEDIUM** (simple base pattern; runaway risk without hard stop/budget guards)
- **Model portability:** **HIGH** (pattern is tool/model-agnostic)

### Recommendation
- **KEEP:** fresh-context loop + persistent progress file as default long-run pattern.
- **STOP:** adopting multi-agent Ralph variants for cron workloads where determinism is priority.
- **TEST:** dual-condition completion gate (completion signal + explicit exit flag) plus hourly budget guard on one long cron.
